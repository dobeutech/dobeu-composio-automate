## Why v5.1 exists

The v5 final consolidated plan (signed off 2026-04-20) is **architecturally sound** — every reviewer-driven structural decision (Linear-as-interaction-only, saga not parallel fan-out, scope-to-5, Supabase ledger as authority) is correct. The Apr 21 execution attempt against Stage 0 surfaced a different class of weakness: the plan's **execution layer** is fragile.

This amendment is additive — it does not modify any v5 plan content. It adds:
- Stricter Stage 0 exit criteria with positive end-to-end evidence requirements
- Environment-dependency declarations in Appendix H subscenario contracts
- Rollback contracts for Stages 1, 2, 3 (Stage 4 already has saga compensation; Stage 5 has the conditional cutover)
- Two reconciled decisions (DTS-1668 ops allocation vs v5 §6, RAG Brain Sync classification)
- Three findings discovered during execution (input-payload contracts, A1 acceptance refinement, Apr 21 checkpoint correction)

v5 plan stays intact as historical reference. Where v5 plan and v5.1 conflict, v5.1 wins — explicit conflict points listed in Appendix N.

## Group A — Stage-Gate Redefinition

### A1 — Stage 0B-minimum exit criteria (REPLACES v5 §0B-min)

**Replacement (full):** Stage 0B-minimum is closed only when each of the 5 core surfaces (Stripe, Typeform, Webflow CMS webhook, Customer.io egress, Linear egress) **has produced**, in response to a signed test webhook:

1. A row in `public.event_ledger` with `status='succeeded'` and a non-null `result_ref`
2. A Linear issue (class-1 `revenue-blocker` or class-4 `milestone` per surface contract)
3. A Datadog event tagged with the surface name and the test `event_key`
4. An R2 archive object at `<surface>/<YYYY-MM-DD>/<event_key>.json` with size > 0

**Acceptance:** five `surface-evidence-<surface>.md` checklist files (plus DLQ self-test) in `Implementation/stage-0-evidence/`, each containing the four artifacts above with timestamps, IDs, and links.

**A1 sub-criteria split (new in v5.1 per item 13):**
- **A1.a** Webhook ingress accepted (HTTP 200 from gateway-webhook URL)
- **A1.b** HMAC verification passed (scenario operations counter incremented past the filter module)
- **A1.c** Downstream side effects produced (Supabase row + Linear issue + Datadog event written)
- **A1.d** R2 archive object written (only applicable to surfaces fronted by the Cloudflare Worker)

A surface PASSES A1 only when all applicable sub-criteria pass. Partial-pass with documented deferral is acceptable per R1 (closing Stage 0 with PARTIAL on a surface is OK if the deferred sub-criterion is captured in `deferred-surfaces.md` with a runbook).

### A2 — Stage 0A exit criteria (REPLACES v5 §0A gate)

All of:
1. `COMPOSIO_MANAGE_CONNECTIONS action=list` returns `default=true, status=healthy` for the 4 named connections (Gmail, Calendar-scoped, Apollo master, Linear)
2. `DOPPLER_SECRETS_NAMES` returns the expected secret name list for project `dobeu-core` config `prd` (use NAMES not GET — see CLAUDE.md § Doppler-via-Composio bans)
3. **A Make scenario successfully reads at least one secret end-to-end via the Doppler-Composio bridge** (functional test, not a literal Doppler-tool call)

### A3 — Sequencing clarification (REPLACES v5 §0 lead "three parallel gates")

The v5 plan's "three parallel gates" framing invited execution drift in the Apr 21 attempt. Replacement dependency shape:

```
0A (blocking)
  └─ 0B-minimum (blocking)
       ├─ 0B-full (parallelizable)
       ├─ 0C       (parallelizable)
       └─ Stage 1  (parallelizable)
```

`0A` and `0B-minimum` are sequential and blocking. After `0B-minimum`, the rest of `0B-full`, `0C`, and `Stage 1` can run in parallel.

### B3 — Stages 1, 2, 3 rollback contracts (ADDS subsection per stage)

#### Stage 1 Rollback Contract

- **Freeze trigger:** Sentry → Datadog → Linear pipeline produces >5 class-3 incidents/hour OR Customer.io JS reports JS errors on `dobeu.dev` or `dobeu.online` >1% of pageviews.
- **Revert action:**
  1. Disable Sentry → Datadog → Linear forwarding (Datadog Issue Template `@linear-critical-incident` set to inactive)
  2. Remove Customer.io JS snippet from Framer Pro head + Webflow head (revert to pre-1.4 state)
  3. Restore Typeform forms `KiSz0R6o`, `tOiaf5BN`, `JQh2CtLK` to direct-to-Make (no UTM / user_id / referrer params)
- **Decider:** user (Jeremy) on class-2 freeze; agent on class-3 freeze with user notify.
- **Smoke test for revert:** open `dobeu.dev` in browser, no CIO `_cio` global; submit Typeform `tOiaf5BN`, no UTM params in landing URL.
- **Class:** class-2 (degraded — surface functional but instrumentation off) or class-1 (revenue-blocker if Stripe path affected).

#### Stage 2 Rollback Contract

- **Freeze trigger:** Tier-4 deprecation removes a scenario still receiving production traffic OR MS Teams `4653925` error rate exceeds 25% post-Azure-AD-fix.
- **Revert action:**
  1. Re-enable any deprecated Tier-4 scenarios via `MAKE_UPDATE_SCENARIO action=update` with `active=true`
  2. For MS Teams `4653925`: revert connection `4528472` to pre-Azure-AD-fix state via Composio reauth
  3. Roll back scenario modernizations on `4654165`, `4652181`, `4458602`, `4653351` to last known-good blueprint snapshot from canonical repo `backups/make/blueprints/`
- **Decider:** agent for revert; user for Azure AD reauth (User Action).
- **Smoke test for revert:** all 4 modernized scenarios fire successfully on test payload; MS Teams `4653925` error rate drops below 14% within 30 minutes.
- **Class:** class-3 (broken automation) for deprecation revert; class-1 if MS Teams blocks customer comms.

#### Stage 3 Rollback Contract

- **Freeze trigger:** Linear egress webhook (3.1) signs invalid HMACs OR DLQ per-family error budget breached (>10/hr per source for 2 consecutive hours) OR GraphQL subscenario `4688114` returns >5% `version_mismatch` errors.
- **Revert action:**
  1. Disable Linear egress webhook in Linear admin → Settings → API → Webhooks (set subscription to inactive) — User Action
  2. Set DLQ per-family circuit breaker to OPEN state (block all DLQ writes from offending source for 1 hour)
  3. Pin `4688114` to last-known-good schema_version; reject `>=2.0.0` calls until investigated
- **Decider:** user on Linear egress disable; agent on circuit breaker.
- **Smoke test for revert:** confirm Linear egress webhook returns 404 from Linear admin; confirm `4688116` receives no new DLQ writes from offending source for 60 minutes; confirm `4688114` calls with current production schema succeed.
- **Class:** class-1 (revenue-blocker for Stripe Kickoff if `4688116` down) for DLQ breach; class-3 for Linear egress; class-2 for GraphQL schema drift.

### B4 — Stripe Kickoff operator-resume scenario (ADDS to Appendix K)

**Scenario:** `<NEW_SCENARIO_ID>` — `stripe-kickoff-operator-resume`
**Status:** draft (waiting for U7 completion; ready for Stage 4.1 deployment)

**Trigger:** Linear webhook on issue comment created, body matches `/^\/resume\b/i` AND issue label includes `stripe-kickoff` (filter to avoid `/resume` noise on unrelated issues)

**Inputs (from Linear webhook payload):**
- `issue_id` — Linear issue containing the saga `correlation_id`
- `comment_id` — for idempotency dedup
- `correlation_id` — extracted from issue body via regex `correlation_id:\s*([a-f0-9-]{36})`

**Side effects:** Re-fire Stripe Kickoff saga steps 3a-3d for edges with `event_ledger.status IN ('failed','locked')` AND `lock_until < now()`. Per v5 plan Appendix K, the four edges are:
- 3a — Create Drive kickoff folder (`GOOGLEDRIVE_CREATE_FOLDER`, idempotency: name `kickoff-<cs_id>` search first)
- 3b — Create Calendar kickoff event (`GOOGLECALENDAR_CREATE_EVENT`, idempotency: `iCalUID = <cs_id>@dobeu`)
- 3c — Customer.io profile + kickoff event (`CUSTOMERIO_IDENTIFY` then `CUSTOMERIO_TRACK`, idempotency: `id = stripe_customer_id`, `event_id = cs_id`)
- 3d — Airtable Customer Pipeline row (`AIRTABLE_CREATE_RECORD` on `tbl8j3gHrIamhLIpd`, idempotency: unique field `stripe_session_id`)

Edges that already succeeded (per `result_ref` in ledger) are skipped on resume — only failed/locked edges re-fire.

**Idempotency:**
- `lock_until` on event_ledger — sets to `now()+5min` before re-firing
- `comment_id`-based dedup: if `comment_id` already in `result_ref.resume_attempts[]`, no-op

**Failure mode:**
- If 2nd resume on same edge also fails: class-2 exception via `4688116`; ledger entry `status='compensated'`; comment back to Linear issue with `/resume failed: <reason>`
- If `correlation_id` cannot be parsed from issue body: class-3 exception, no-op

**Environment Dependencies:**
- Make scenario env vars: `STRIPE_KICKOFF_RESUME_LOCK_MINUTES` (default `5`)
- Composio connections: linear (`4727177`), supabase, stripe (`4526971` or successor)

## Group C — Decision Reconciliation

### C1 — DTS-1668 vs v5 §6 reconciliation

**Decision:** Option (ii) — permit a parallel **v5.1a Path Alpha** track running in the gap between Stage 0 close and 5-core boring, with explicit gate "v5.1a cannot consume Stage 0-5 capacity."

**Rationale:** the 17% capacity diversion (3 hr/wk against 18 hr/wk) is small. StatMiner is a strategic surface (NL→SQL is differentiated). The explicit gate makes contention visible rather than implicit. Path Alpha conditionally suspends on any class-3 exception in 5-core lasting ≥48 hours.

**Effective date:** 2026-04-25

**Implications:**
- DTS-1668 allocation table: unchanged (StatMiner 3 hr/wk slot stays)
- v5 plan §6: cross-reference points to "v5.1a Path Alpha permitted with gate" in this amendment
- Gate enforcement: Path Alpha track suspends on any class-3 in 5-core ≥48h

**Linear comment drafts staged for sending** (orchestrator sends in a follow-up step; see `Implementation/v5.1-amendment-staging.md` § C1 for full text):
- DTS-1668: reconciliation note + link to this file
- DTS-1671: gate language + link to this file

### C2 — RAG Brain Sync (Stage 4.7) classification

**Decision:** Option (ii) — agent-driven carve-out.

**Rationale:** workload is genuinely agent-shaped (LLM embedding + decision-driven file selection). Forcing into Make has high friction (embedding API calls, vector logic, file selection branching) and low benefit (weekly cadence, no real-time requirements). Carve-out + Datadog observability gives monitoring without architectural compromise.

**Implementation contract:**
- Runs in Composio Remote Workbench, NOT a Make scenario
- Schedule: weekly Sunday 03:00 UTC via scheduled-tasks plugin
- Observability: agent forwards logs to Datadog via Datadog HTTP intake API. **Per CLAUDE.md Datadog reliability rule, use direct HTTP with both API key + App key headers, NOT Composio's Datadog connector.**
- Tags: `service:rag-sync, environment:stage4`
- Failure mode: scheduled task failure forwards Datadog event with tag `rag_brain_sync.failed` → Datadog monitor opens Linear class-3 exception via `4688116`
- **Carve-out documentation:** future agents reading the v5 plan should NOT expect Stage 4.7 to follow Make scenario standards (no Appendix H contract, no idempotency key per the standard pattern)

- Failure mode for each common malformed-input case

**Acceptance:** scenarios with undocumented contracts as of 2026-04-25 (per `Implementation/stage-0-evidence/deferred-surfaces.md`) all have input-payload contract sections in Appendix H before Stage 1 starts. Specifically: `4688746` (DLQ), `4653306` (Typeform Submit), `4653307` (Typeform Rate), `4653351` (Webflow Review Published).

### Item 13 — A1 acceptance refinement (HMAC vs tri-write distinction)

Already integrated as A1.a-d sub-criteria above. The original binary 4-artifact A1 is replaced with 4 sub-criteria, each independently checkable.

### Item 14 — Apr 21 checkpoint correction

**Trigger:** Task 1 finding. The Apr 21 checkpoint stated "tri-write blocker is now isolated to HMAC secret mismatch at module 2." That conclusion was incomplete: HMAC was ONE blocker; the undocumented BasicRouter input contract is another, independent blocker that fixing HMAC does not address.

**Pin to Appendix N:** Apr 21 checkpoint conclusion was incomplete. HMAC fix was necessary but not sufficient for tri-write. The second blocker (scenario input-payload contract) is addressed in v5.1 amendment item 12.
